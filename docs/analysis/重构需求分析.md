### 1.2 与架构文档 v1.4 的关键差距

| # | 差距项 | 架构要求 | 现状 | 影响 |
|---|--------|---------|------|------|
| G1 | **匹配异步化** | 通过 `matching_queue` 异步处理，写 `mq_message_log`，发送 `notification_queue` | 同步处理；无 RabbitMQ Producer/Consumer；无 `MqMessageLog` 实体写入 | 无法承载高并发，无失败重试 |
| G2 | **规则过滤层** | 硬性条件（过敏、大体型/小公寓、对方有小孩 ≠ 用户无小孩等） | **未实现**，匹配阶段直接用向量相似度 | 误匹配风险大，违背"减少退回"核心价值 |
| G3 | **向量维度一致性** | 架构标 4096 维；schema.sql 写 2000 维；Embedding 模型 Qwen3-Embedding-8B 实际为 4096 | 三方不一致 | **数据库插入会报错或精度下降** |
| G4 | **向量索引** | 架构标 HNSW | 实际用 ivfflat | ivfflat 需预训练且召回率低于 HNSW |
| G5 | **GIN 元数据过滤** | 架构要求先用 GIN 过滤 metadata (species/size/...) 再 HNSW | 现状只做向量排序，无 metadata 过滤 | 召回候选不准且慢 |
| G6 | **意图识别** | 架构要求"AI 意图识别" | 现状为关键词字符串匹配 | 容易误识别，多轮对话差 |
| G7 | **RAG 分块策略** | 架构要求 chunk_size=512/overlap=50 | 未做知识库分块入库管理；无 ETL | 知识库只能整段入库，召回质量差 |
| G8 | **匹配理由生成** | 架构允许多种实现 | **每条结果都同步调用 LLM**，未走异步队列 | 匹配一次耗时高、费用高 |
| G9 | **匹配权重** | 架构 6 个维度（性格 30%/体型 25%/经验 20%/活动 15%/健康 10%） | `application.yml` 仅有 5 个权重（vector/species/size/age/experience），缺活动水平 + 健康 | 与架构不一致 |
| G10 | **认证安全** | 架构未细化 | `/api/pets/**`、`/api/chat/**` 直接 permitAll | 应用、领养人画像未保护，存在越权 |
| G11 | **目录结构** | 架构要求 `dto/request/`、`dto/response/`、`model/enums/`、`ai/` 子包 | 现状 `dto/` 平铺、无 `enums/`、无 `ai/` 包 | 不符合架构规范 |
| G12 | **错误码体系** | 架构要求 `ErrorCode` 枚举 | 已实现，但 `Result.error(String)` 仍存在；`GlobalExceptionHandler` 仍用 try/catch 倾向 | 与项目规则"少 try/catch、统一 Result"不完全一致 |
| G13 | **前后端协议差异** | 架构：`/api/match/recommend` 为 POST body；前端：GET `/match/recommend?topK` | 接口方法不一致 | 前端调用可能与后端签名不匹配 |
| G14 | **聊天会话历史未加载** | 架构要求支持会话列表 + 历史消息 | 前端 `Chat.tsx` 中 `loadSessions` 实际为空 | 历史功能形同虚设 |
| G15 | **前端 RTK Query 未使用** | 架构：Zustand + RTK Query | 仅安装未使用，API 调用全部走裸 axios | 重复样板、缓存缺失 |
| G16 | **无 CORS/跨域统一配置** | 架构未明确，但前后端分离必要 | 仅 `WebConfig`，未确认是否配置 CORS | 浏览器跨域请求可能失败 |
| G17 | **Embedding 模型维度硬编码** | 架构允许调整 | `VectorSearchService` 写死 dimension=2000 | 与真实模型不一致 |
| G18 | **API 文档缺失** | 架构未明示但应具备 | 无 Swagger/OpenAPI | 联调与维护成本高 |
| G19 | **单元测试缺失** | 架构未明示但应具备 | `src/test/java/` 为空 | 重构缺乏安全网 |
| G20 | **JPA/MyBatis 混合** | 架构选 MyBatis-Plus | 现状既有 MP 也有 `JdbcTemplate` 跑原生 SQL | 向量检索可接受，但应集中封装 |

## 二、重构目标

1. **对齐架构**：所有缺口（G1–G11）逐项落地，使实现 100% 反映架构 v1.4。
2. **夯实核心**：匹配系统具备 **异步、规则过滤、可观测、可降级** 四大能力，确保生产可用。
3. **提升质量**：补齐安全、错误处理、测试、文档、CORS 等"非功能"短板。
4. **保持设计风格**：前端不做大改，按项目规则保持暖橙 + 米白 + 圆角风格。
5. **遵循项目规则**：
   - 后端不写大量 try/catch，统一用 `GlobalExceptionHandler` + `Result`；
   - 仅做必要修改，不引入新框架或大重构。

## 三、功能性需求（FR）

### FR-1 用户与权限
- **FR-1.1** 维持现有注册/登录/登出/JWT 刷新逻辑；密码 BCrypt 加密。
- **FR-1.2** 引入 `UserRole`、`ShelterType`、`PetSpecies`、`PetSize`、`PetGender`、`HousingType`、`ActivityLevel`、`ApplicationStatus` 枚举类（取代裸字符串 `"ADOPTER"` 等）。
- **FR-1.3** Security 配置：默认 `/api/auth/**` permitAll，其余必须带 JWT；按角色 `@PreAuthorize` 限制：
  - `/api/applications/*/review`、`/api/pets/**` POST/PUT/DELETE → `SHELTER` / `ADMIN`
  - `/api/stats/platform` → `ADMIN`
  - `/api/match/recommend`、`/api/applications/my` → 已登录任意角色

### FR-2 宠物档案
- 保持现有 CRUD + 分页 + 多条件筛选。
- 新增 `photos` 字段写入支持（已有 List<String>）。
- 创建/更新后**异步**触发 embedding 重算（见 FR-5）。

### FR-3 救助站
- 现有 CRUD 保留；新增"审核工作流"接口（ADMIN 调用）。

### FR-4 领养人画像
- 维持现有 CRUD。
- 创建/更新后**异步**触发 embedding 重算。

### FR-5 AI 智能匹配（核心重构）
- **FR-5.1 异步化**：
  - `POST /api/match/recommend` 立即返回 `taskId`，匹配在 `matching_queue` 消费者中执行。
  - 落库 `mq_message_log`，状态 PENDING → SUCCESS / FAILED；记录 retryCount / errorMessage。
  - 提供 `GET /api/match/history?taskId=...` 查询状态；前端轮询或 WS 推送（先做轮询）。
- **FR-5.2 三步流水线**：
  1. **召回**：以领养人向量查 pgvector（HNSW，Top-K=20~50）。
  2. **规则过滤**：硬性条件——过敏冲突 / 大型犬+小公寓 / 用户无小孩但宠物要求有小孩家庭等。
  3. **精排打分**：六维加权（性格 30% / 体型 25% / 经验 20% / 活动 15% / 健康 10% / **向量相似度作为先验** 0% 或独立维度）。
- **FR-5.3 匹配理由生成**：从同步改为异步 + 缓存；同一 (adopterId, petId) 仅生成一次，结果写入 `adoption_application.matching_reasons`。
- **FR-5.4 通知**：`notification_queue` 异步通知申请人（站内信 / WebSocket，先落库即可）。

### FR-6 RAG 智能问答
- **FR-6.1 知识库 ETL**：提供 admin 接口支持上传 Markdown/文本，按 512/overlap=50 切块，批量 embedding 入库。
- **FR-6.2 检索策略**：先 GIN 过滤 metadata (category)，再 HNSW 向量召回 Top-K=5。
- **FR-6.3 意图识别**：将关键词规则替换为可选项——优先走 LLM 分类（轻量 prompt），关键词规则保留作为降级 fallback。
- **FR-6.4 多路召回**：知识库片段 + 宠物档案片段 + 对话历史一并送入 LLM。
- **FR-6.5 流式输出**：可选优化，先按非流式实现，标记为 P2。

### FR-7 领养申请 + 回访
- 现有功能保留；新增：在状态变更时落 `notification_queue`。
- 回访提醒：根据 `next_follow_up_date` 提供 `GET /api/follow-ups/upcoming` 给救助站看板。

### FR-8 数据统计
- 维持现有；新增"匹配转化漏斗"：发起匹配 → 提交申请 → 审核通过 → 完成领养。

### FR-9 系统能力
- FR-9.1 引入 **OpenAPI/Swagger**（springdoc-openapi）。
- FR-9.2 引入 **Redis 缓存**：匹配结果（短 TTL）、Embedding（长期）、会话列表。
- FR-9.3 **CORS** 显式配置：允许 `http://localhost:5173` 等开发源。
- FR-9.4 **统一日志**：traceId + MDC，便于排查。
- FR-9.5 **Embedding 维度修正**：统一为 4096，重建 schema（提供 migration 脚本）。

## 八、验收标准

### AC-1 数据一致性
- 所有 Embedding 表 schema 维度统一为 4096。
- 索引全部为 HNSW。
- 三处 schema（schema.sql / 迁移脚本 / 应用启动检查）一致。

### AC-2 匹配异步化
- `POST /api/match/recommend` < 200ms 返回 `taskId`。
- 消费者在 5s 内完成 1 条匹配。
- `mq_message_log` 表中能看到对应记录，状态正确流转。

### AC-3 规则过滤生效
- 单元测试：用户标注过敏 → 不返回猫/狗中相关种类。
- 用户住房为 SMALL_APARTMENT → 大型犬权重为 0。

### AC-4 RAG 双路召回
- 知识库有 "领养流程" 类目 → 用户问"流程"应优先召回该类。
- 召回 SQL 包含 `metadata @> '{...}'::jsonb` 过滤。

### AC-5 安全
- 未带 Token 调用 `/api/match/recommend` 返回 401。

## 九、关键决策（已与用户确认）

### D1. Embedding 模型
- **当前模型**：`Qwen/Qwen3-Embedding-8B`（4096 维，硅基流动）
- **可替换性**：设计上保持可替换，但需要：
  - 数据库列改 `VECTOR` 不指定维度（或用宽列 + 冗余 dimension 字段）
  - 应用层从 `EmbeddingModel` 动态读取维度，禁止硬编码
  - 模型切换时一次性脚本重建所有向量
- **数据库建议**：`embedding VECTOR`（不指定维度上限，pgvector 默认 16000），`dimension INT` 单独存
- **配置文件**：`embedding.model` + `embedding.dimension` 均外置

### D2. 异步匹配结果返回方式
- **MVP 采用轮询**：间隔 1.5s，最多 10 次（约 15s），超时后提示"查看历史"
- **预留 WebSocket 升级路径**：
  - `MatchHistory` 表 + `taskId` 是统一抽象，未来切 WS 只改前端订阅逻辑
  - 后端事件总线未来可用 RabbitMQ Fanout → STOMP Broker Relay
- **轮询接口**：`GET /api/match/history/{taskId}` 返回 `{status, score, pets[], reasons, errorMessage}`

### D3. LLM 成本控制策略
- **成本量级估算**（DAU=1000）：
  - 匹配理由：≈¥1080/月
  - RAG 问答：≈¥1800/月
  - Embedding：可忽略
  - **合计 ≈ ¥3000/月**
- **降本手段**（按优先级）：
  1. **缓存**：同 (adopterId, petId) 复用理由（Redis TTL 7 天），降 70%
  2. **异步化**：用户无需等待理由，节省交互成本
  3. **模板降级**：理由改为模板拼接，**零 LLM 成本**，评分仍可用 LLM
  4. **分级模型**：分类用 `qwen-turbo`、理由用 `qwen-plus`、RAG 用 `qwen-max`
  5. **用户限流**：每天最多匹配 N 次
- **Embedding**：硅基流动按 token 计费，约 ¥0.0005/次，可忽略
