## 二、当前项目问题清单

### 2.1 功能性缺陷（Functional Defects）

#### 🔴 F-01：前端 ↔ 后端枚举协议不一致

**现象**：用户提交领养人画像时，前端 [MyProfile.tsx](file:///E:/study/PawMatch/PawMatch1/frontend/src/pages/MyProfile.tsx#L11-L23) 直接传中文枚举字面量：

```ts
housingType: '公寓',
petExperience: '有一些经验',
activityLevel: '适量活动',
```

而后端 [HousingType.java](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/model/enums/HousingType.java) 等枚举 code 是 `SMALL_APARTMENT / APARTMENT / HOUSE_WITH_YARD`。

**影响**：
- DB 实际写入字符串 `"公寓"` 等非枚举值；
- `HardRuleFilter.R2` 的 `equalsIgnoreCase("SMALL_APARTMENT")` 永远 false → R2 规则失效；
- `MatchingService.computeActivityScore` `equalsIgnoreCase("LOW")` 永远 false → 活动水平打分恒为 0.5（兜底值）；
- 经验维度同理 → 整体匹配分失真。

**修复方向**：
1. 前端 MyProfile 表单层加 enum 映射（参考 [PetList.tsx](file:///E:/study/PawMatch/PawMatch1/frontend/src/pages/PetList.tsx#L36-L51) 已有的映射方式）；
2. 后端在 `AdopterProfileService.createOrUpdateProfile` 增加 `HousingType.fromCode()` 兜底，未匹配则抛 `BusinessException(BAD_REQUEST)`。

#### 🔴 F-02：Home 页"为你推荐"功能误导

**现象**：[Home.tsx](file:///E:/study/PawMatch/PawMatch1/frontend/src/pages/Home.tsx#L77-L82) 文案写 "基于你的偏好，AI 精选匹配"，但实际调的是 [PetService.getRecommendedPets(7)](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/PetService.java#L151-L156) —— 该方法仅按 `createdAt DESC` 排序取最新 7 条 **AVAILABLE** 宠物，**与用户画像完全无关**。

**影响**：
- 未登录用户也显示"为你推荐"——但根本没画像，谈不上推荐；
- 即使登录用户看到的也是最新发布宠物，匹配逻辑完全缺席；
- 与宣传卖点"AI 智能匹配"不符，对核心价值有损害。

**修复方向**：登录用户走 `matchApi.recommend(topK)`；未登录用户走"最新宠物" + 改文案为"最新待领养"。

#### 🔴 F-03：领养人画像缺 `implicit_preferences` 字段（隐式偏好反哺）

**现象**：架构文档 [FR-3](file:///E:/study/PawMatch/PawMatch1/plans/03-需求分析.md) 要求"画像更新后异步触发 embedding 重算"，已通过 P0-2 修复；但架构要求的"行为反哺"（浏览事件、停留时长、收藏）**完全没有数据采集**——前端无埋点 SDK，后端无 `browse_event` / `implicit_feedback` 表。

**影响**：用户画像仅来源于主动填表，对未填画像的"隐性偏好"无法感知 → 推荐召回率受限于用户主动行为。

#### 🟠 F-04：救助站审核工作流前端入口缺失

**现象**：`ShelterService.verifyShelter` 与 `POST /api/shelters/{id}/verify` 已实现，但前端 [App.tsx](file:///E:/study/PawMatch/PawMatch1/frontend/src/App.tsx) 无 `/admin` 路由，无审核 UI。

**影响**：管理员角色无实际工作台，verify API 形同虚设。

#### 🟠 F-05：回访提醒功能无对应定时任务

**现象**：`follow_up_record.next_follow_up_date` 字段已建表，但后端无 `@Scheduled` 定时扫描，无 `GET /api/follow-ups/upcoming` 真正调用。

#### 🟡 F-06：聊天会话标题未自动生成

**现象**：[ChatService.createSession](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/ChatService.java#L87-L95) 硬编码 `title = "新对话"`，从不更新。

**影响**：历史侧栏所有会话都显示"新对话"，用户无法区分。

#### 🟡 F-07：注册流程未触发 default profile 创建

**现象**：用户注册后没有同步创建 `adopter_profile` 行，必须访问"我的画像"页面才会触发首次创建。新注册用户直接调匹配接口必然空召回。

### 2.2 性能瓶颈（Performance Bottlenecks）

#### 🔴 P-01：Embedding 维度与 DB Schema 不一致

| 项 | 现状 |
|----|------|
| DB 列 | `pet_embedding.embedding VECTOR(2000)` |
| 应用配置 | `embedding.dimension: 2000` ([application.yml:76](file:///E:/study/PawMatch/PawMatch1/backend/src/main/resources/application.yml#L76)) |
| 实际模型 | `Qwen/Qwen3-Embedding-8B` 4096 维 |

**风险**：
- 当前模型若返回 4096 维向量，插入 `VECTOR(2000)` 会 PG 报错；
- 当前 P0-2/P0-3 修完后所有新增向量会**全部插入失败**（因为 EmbeddingService 通过 `getDimension()` 动态探测，得到 4096，写入 2000 列溢出）。

**实测**：本地 `mvn test` 不接真实 EmbeddingService，但 `EmbeddingServiceTest` 通过 mock 模拟了 4096；意味着**生产环境首次写入向量就会爆**。

**修复方向**：V3 Flyway 迁移把列改 `VECTOR`（不指定维度）或 `VECTOR(4096)`；同步 application.yml。

#### 🟠 P-02：向量召回 SQL 无 metadata GIN 预过滤

**修复状态**：P1-8 已加 `metadata @> '{...}'::jsonb`，但：
- 仅对 `category` 生效，其他 metadata 字段（`housingType`、`activityLevel`）未使用；
- [VectorSearchService.searchSimilarPets](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/VectorSearchService.java#L107-L116) 完全没用到 metadata 过滤——它走的是 `pet_embedding`，没有 `WHERE metadata @> ...` 子句。

**瓶颈**：候选集全表扫描后 HNSW 排序，Top-K=10 时延迟 P99 估计 200-500ms（10 万级数据下）。

#### 🟠 P-03：匹配理由同步生成阻塞 Consumer

**修复状态**：P2-10 已通过 [AsyncReasonExecutor](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/AsyncReasonExecutor.java) 把理由生成移到独立线程池。但：
- LLM 调用仍是同步 `chatClient.call().content()`，每条 ~1-3s；
- 线程池容量 = `min(16, CPU*4)`，超过即排队；
- 没有超时控制，单条 LLM 卡死会拖垮整个池。

#### 🟡 P-04：StatsService.getMatchingFunnel 全表扫描

[StatsService.java#L202-L246](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/StatsService.java#L202-L246) 每次调用都跑 4 次 `selectCount`，无缓存。DAU 1000+ 时漏斗接口会被频繁刷，数据库压力大。

**修复方向**：Redis 缓存 5 分钟，或物化视图。

#### 🟡 P-05：VectorSearchService 用 JdbcTemplate 走原生 SQL

[VectorSearchService.java#L107-L116](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/VectorSearchService.java) 用 JdbcTemplate 跑 `SELECT ... ORDER BY embedding <=> ?::vector LIMIT ?`，与项目主体 MyBatis-Plus 风格混用。

**影响**：
- 维护成本高（两套 SQL 风格）；
- 失去 MP 的分页、动态条件拼接能力。

### 2.3 用户体验问题（UX Issues）

#### 🔴 UX-01：401 直接跳登录页，无提示

[request.ts#L47-L52](file:///E:/study/PawMatch/PawMatch1/frontend/src/api/request.ts#L47-L52) Token 失效直接 `window.location.href = '/login'`，丢失当前页路径与上下文，用户被踢回登录页后必须重新导航。

**修复方向**：弹 Modal "登录已过期，是否重新登录？"，确认后再跳。

#### 🟠 UX-02：匹配结果轮询期间无进度反馈

[match.ts](file:///E:/study/PawMatch/PawMatch1/frontend/src/api/match.ts#L74-L101) 1.5s 轮询直到 SUCCESS / FAILED。匹配平均耗时 5-10s，用户看到的是 "loading..." 10 秒——**没有任何进度条或骨架屏**。

**修复方向**：
- 进度提示 "AI 正在召回候选..." → "正在打分..." → "生成理由..."；
- 或改 SSE 推送阶段事件。

#### 🟠 UX-03：聊天界面首屏问题

[Chat.tsx](file:///E:/study/PawMatch/PawMatch1/frontend/src/pages/Chat.tsx#L25-L46) 未登录也能看到 WELCOME 消息，但发送时会失败——错误处理是静默 toast "AI 助手暂时无法响应"，用户不知道为什么。

#### 🟠 UX-04：宠物详情页无相似推荐

[PetDetail.tsx](file:///E:/study/PawMatch/PawMatch1/frontend/src/pages/PetDetail.tsx) 打开后只展示一只宠物，无"类似宠物" / "用户也喜欢" 区域。

**影响**：跳出率高，转化漏斗在"详情 → 申请"步骤损失大。

#### 🟡 UX-05：救助站工作台缺少批量操作

[ShelterDashboard.tsx](file:///E:/study\PawMatch\PawMatch1\frontend\src\pages\ShelterDashboard.tsx) 宠物列表逐个操作，没有批量上下架、批量审核。

#### 🟡 UX-06：搜索功能弱

[PetList.tsx](file:///E:/study/PawMatch/PawMatch1/frontend/src/pages/PetList.tsx) 筛选只有物种/体型/年龄/性别/健康，无关键词搜索（按名字、品种）。`PetQueryRequest` 也无 keyword 字段。

### 2.4 潜在技术风险（Technical Risks）

#### 🔴 R-01：API Key 明文提交

[application.yml:41](file:///E:/study/PawMatch/PawMatch1/backend/src/main/resources/application.yml#L41) 与 [application.yml:50](file:///E:/study/PawMatch/PawMatch1/backend/src/main/resources/application.yml#L50) 把硅基流动 + DashScope API Key 直接写在 yml 并 commit 进仓库：
```yaml
api-key: sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx   # 占位示例（已脱敏）
```

**风险**：任何人克隆仓库即可盗用配额。生产用 env 变量未覆盖。

#### 🔴 R-02：JWT Secret 默认值仍生效

[application.yml:70](file:///E:/study\PawMatch\PawMatch1\backend\src\main\resources\application.yml#L70)：
```yaml
jwt:
  secret: your-jwt-secret-key-at-least-256-bits-long
```

若生产未通过环境变量覆盖，会用默认值签发 Token——任何拿到源码的人都能伪造 admin Token。

**修复方向**：启动校验 `JWT_SECRET` 必须 ≥ 256 bit 且 ≠ 默认值，否则 fail-fast。

#### 🟠 R-03：CI 未跑集成测试

[.github/workflows/ci.yml:60](file:///E:/study/PawMatch/PawMatch1/.github/workflows/ci.yml#L60) 跑的是 `mvn verify`，但 `application-integration.yml` 实际是 H2 + 排除 Redis/RabbitMQ 的"伪集成"（见 [application-integration.yml](file:///E:/study/PawMatch/PawMatch1/backend/src/test/resources/application-integration.yml)）——所谓 `MqMessageLogServiceIT` 不会真正连 RabbitMQ。

**风险**：向量写入 / MQ 重试 / Redisson 锁的代码改动没有运行时验证。

#### 🟠 R-04：可观测性基础设施缺失

| 维度 | 现状 |
|------|------|
| Metrics | 无 Micrometer / Prometheus 端点 |
| Logging | 无 traceId MDC，无结构化日志 |
| Tracing | 无 OpenTelemetry |
| Health | `/actuator/health` 未配置（[Dockerfile:49](file:///E:/study/PawMatch/PawMatch1/backend/Dockerfile#L49) 引用了但 yml 没启用） |

**影响**：生产出问题无法快速定位是 API 慢还是 DB 慢还是 LLM 超时。

#### 🟠 R-05：缺失限流

无任何限流措施：
- LLM 调用：用户可无限触发匹配 → 单个用户可刷爆 Token 配额；
- 注册接口：无 CAPTCHA，无 IP 限频；
- 申请提交：仅靠应用层"已存在申请"检查防重，并发下可能产生重复。

#### 🟠 R-06：JWT Filter 不主动 reject 失败 Token

[JwtAuthenticationFilter.java#L60](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/security/JwtAuthenticationFilter.java#L60) 对解析失败的 Token 仅 log warn 然后放行——后续 filter 又会放行（因为没 SecurityContext）→ 用户调受保护接口得到 401 而不是友好的"Token 已过期"。

#### 🟡 R-07：EmbeddingService.getDimension() 首次调用探测

[EmbeddingService.java#L41-L49](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/EmbeddingService.java#L41-L49) 首次 `embed("dim-probe")` 会真实调一次 LLM 并付费 ¥0.0005。在生产启动期或测试环境会重复探测，浪费 token 且可能因网络抖动失败。

#### 🟡 R-08：MyBatis + JdbcTemplate 混用

[VectorSearchService](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/VectorSearchService.java) 与 [HybridSearchService](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/HybridSearchService.java) 走 JdbcTemplate；其他 Service 走 MP。两套风格，SQL 散落 service 层。

#### 🟡 R-09：HardRuleFilter.R5 "NONE_" 协议是约定

[HardRuleFilter.java#L96-L106](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/service/HardRuleFilter.java#L96-L106) 通过 `preferredPetSize` 列表中包含 `"NONE_LARGE"` 字符串视为排斥。**该协议无任何文档或前端实现**——靠前端枚举映射时容易遗漏。

#### 🟡 R-10：MQ 消息无死信队列（DLQ）

5 个队列（matching / notification / chat / adopter_vector / pet_vector）都未配置 DLX。失败 3 次后消息直接丢弃，运维无法人工介入。

## 四、SWOT 综合分析

### 4.1 优势（Strengths）

1. **RAG 智能问答技术深度领先竞品** —— 混合检索 + 意图识别 + 模板降级全套实现，竞品全部缺失或简陋；
2. **6 维匹配打分 + 规则过滤** —— 比"关键词匹配"先进，与宠知道的 27 维仍有差距但方向正确；
3. **Cloud Native 部署就绪** —— Helm + KEDA + Istio + ArgoCD 文档齐全，毕设/演示加分项；
4. **设计差异化** —— 暖橙 + 米白"治愈系"风格在领养场景有情感共鸣；
5. **代码工程化好** —— 68 个单测、IdempotentConsumerHelper、Result 统一返回、Flyway 版本化迁移。

### 4.2 劣势（Weaknesses）

1. **关键功能未上线** —— 行为采集、回访提醒定时、管理后台 UI；
2. **前后端协议不一致**（F-01）—— 直接导致 R2/R3 规则失效，是 P0 级阻断；
3. **向量维度不匹配**（P-01）—— 生产首次写入就会爆；
4. **API Key / JWT Secret 明文**（R-01, R-02）—— 安全红线；
5. **无可观测 / 无风控 / 无移动端** —— 距生产可用差明显。

### 4.3 机会（Opportunities）

1. **AI 大模型平民化** —— Qwen3-Embedding 与 Qwen-plus 成本持续下降，RAG 体验会更好；
2. **多模态模型成熟** —— 可补品种识别、健康评估（对齐宠知道/Pawzzle）；
3. **政策合规推动集中化** —— 博研报告指出 2025 年 32 家小平台因合规要求退出，头部平台受益；
4. **社区 + 长尾运营** —— Petfinder 强在社区，PawMatch 可借暖色风格做差异化社区。

### 4.4 威胁（Threats）

1. **它基金** 公益背景 + 政策资源（民政部社区试点）—— 政策红利难撼动；
2. **宠知道** 27 维 AI + 阿里云合作 —— 技术 + 渠道双壁垒；
3. **Pawzzle** AI Agent 概念前卫 —— 早期占心智；
4. **LLM API 涨价或限流** —— 月度 ¥300 成本在小流量下可承受，规模化后压力陡增；
5. **数据合规** —— 用户数据 + 宠物数据 + 地理位置，未来《个保法》合规成本。

## 六、与竞品差距的关键决策点

> 以下为基于对比分析得出的建议方向（非产品需求结论，请按实际业务取舍）。

### 决策点 D1：要不要做多模态识别？

- **跟随派**：补品种识别、健康评估，对齐宠知道/Pawzzle；
- **差异化派**：聚焦 RAG 问答深耕，做"领养知识专家"差异化；
- **建议**：先做 RAG 深耕（已有 90% 实现），多模态作为下一阶段增量。

### 决策点 D2：要不要做自然语言对话画像采集？

- **做**：对齐 Pawzzle 的"对话式画像"，降低用户填表门槛；
- **不做**：保持当前表单式 + 行为反哺；
- **建议**：行为反哺优先（影响所有人），对话式作为高级选项。

### 决策点 D3：要不要做社区？

- **做**：对齐 Petfinder 强壁垒，但运营成本高；
- **不做**：聚焦匹配 + 问答，依赖第三方社区引流；
- **建议**：MVP 不做，但保留接口（已在 [ChatSession](file:///E:/study/PawMatch/PawMatch1/backend/src/main/java/com/petadopt/entity/ChatSession.java) 等）。

### 决策点 D4：要不要做风控 / 反欺诈？

- **做**：领养欺诈（虐待、贩运）是行业痛点；
- **不做**：纯匹配平台，依赖救助站人工把关；
- **建议**：先做"行为画像异常预警"（多次申请被拒、IP 异常），不做生物识别。

## 七、结论

PawMatch **技术架构扎实、Cloud Native 与异步流水线成熟**，但**功能性完整度仅 68%**，距离生产可用还需完成 10 项左右核心改动。其中 **F-01（前后端枚举）、P-01（向量维度）、R-01/R-02（密钥管理）** 是必修项，**3 天内可闭环**。

在竞品维度，**RAG 智能问答、Cloud Native 部署、设计风格** 是显著优势；**多模态识别、自然语言对话、社区、履约风控** 是显著短板。**产品定位上建议聚焦"AI 智能匹配 + RAG 问答"双引擎**，不与大平台拼覆盖广度，而做"领养决策专家"的差异化心智。
